## Work log

### Local dev CORS question — answered, not a bug

Arinzechukwu asked whether hitting the same generic error running `apps/web`
locally was expected, now that production's `CORS_ORIGIN` is locked to
exactly `https://app.schoolkit.ng`. Confirmed yes, and not a regression:
`.env.example`'s documented local setup is local `apps/web` → local
`apps/api` (`pnpm dev:api`), which `main.ts`'s `CORS_ORIGIN ?? "http://
localhost:3001"` fallback already covers with no env var needed. Pointing
local `NEXT_PUBLIC_API_URL` at the production API was never going to work
via CORS — before yesterday's fix it failed for the wrong reason, after it
fails for the (correct, now-locked) real reason. One nuance surfaced while
answering this that mattered a lot for the next investigation: login/signup
route through a **server-side** Next.js proxy
(`apps/web/src/app/api/auth/[...auth]/route.ts`), never subject to browser
CORS at all — only the client-direct endpoints (guardians, students,
invoices, etc.) hit the CORS boundary.

### Production "can't create a new school" — a second, unrelated bug

Reproduced directly against the real domain this time (not curl-to-Fly, since
this flow is server-proxied): `POST https://app.schoolkit.ng/api/auth/
signup-owner` → `500`, empty body. `vercel logs --expand` gave the real
stack trace immediately:

```
[TypeError: fetch failed] { [cause]: [Error: getaddrinfo ENOTFOUND
api.placeholder.example.com] }
```

`school-kit-web`'s `NEXT_PUBLIC_API_URL` had a placeholder value baked into
the deployed bundle — `vercel env pull` showed an empty string for the
*current* dashboard value, but Next.js inlines `NEXT_PUBLIC_*` vars at
**build time**, so whatever the currently-live build was compiled with is
what mattered, and that was the placeholder host. Confirmed via `vercel env
ls`: this is the *only* env var configured for `school-kit-web` at all —
no Sentry DSN, nothing else.

Fixed: `vercel env rm`/`add NEXT_PUBLIC_API_URL` (production + preview) →
`https://school-kit-api.fly.dev/api/v1`, then a forced `vercel --prod`
rebuild (updating the dashboard value alone doesn't touch an already-built
bundle). Hit a Vercel CLI quirk deploying from within `apps/web` (the
linked project's dashboard Root Directory setting double-joins with the
invocation cwd, producing a nonsense `apps/web/apps/web` path) — worked
around it by deploying from the true repo root with `VERCEL_ORG_ID`/
`VERCEL_PROJECT_ID` set explicitly rather than relying on the `.vercel/`
link file's location. Re-ran the exact same reproduction post-deploy:
`201 Created`, real session cookie. Cleaned up the verification school via
the same production-container script pattern from yesterday.

**Exposure window and real-impact check, done the same evidence-based way
as yesterday's CORS entry:** the env var key is "created 38 days ago" per
Vercel — an upper bound, not a confirmed continuous-bad-value duration
(Vercel's CLI doesn't expose value-change history). Queried every school in
production for real impact instead of guessing: **every single row is a
`Smoke Test School` artifact** from `scripts/smoke-test.sh`'s own
direct-to-Fly-API calls (which bypass the broken proxy entirely — why
smoke tests kept passing the whole time). Zero genuine signups exist
anywhere in the database. Combined with this project being pre-pilot, that
closes the question cleanly: no real customer was ever affected, only
Arinzechukwu's own manual QA — which is exactly what surfaced it.

Logged in `docs/deferred.md` (marked `[x]` FIXED), same discipline as
yesterday's two entries.

### Incidental finding: production accumulates smoke-test schools forever

The real-impact query above surfaced a separate, lower-priority gap:
`scripts/smoke-test.sh`'s signup call creates a genuine `School`+owner
`User` row on **every** `deploy-staging.yml` run and never deletes it —
production has been silently accumulating one more every deploy, with no
cleanup path (no `DELETE /schools/:id` API exists). Logged as a new,
separate `[ ]` open item in `docs/deferred.md` — not fixed, just flagged,
per the user's explicit "log it, don't fix it now" instruction.

Both of today's `deferred.md` entries shipped together in PR #105
(`docs/prod-smoke-test-school-cleanup`, merged `93c7c8d`), same
plan-first → build → PR → CI → merge → fresh-clone-verify rhythm as every
code slice, even though this was docs-only.

### Sentry wiring for `apps/web` — closing the "zero automated detection" gap

Three production incidents in two days (Fly cold-start, CORS, wrong API
URL) were all found only by manual testing — none would have paged anyone.
User's instruction: don't defer fixing this one, wire it now. Checked
whether a Sentry project for `school-kit-web` already existed before doing
anything — no access to Sentry myself, so asked the user to check directly
rather than guessing or unilaterally provisioning a new project. It
existed, unconfigured (DSN provided by the user).

Confirmed the exact env var name expected — `NEXT_PUBLIC_SENTRY_DSN`,
consistently read across `sentry.client.config.ts`, `sentry.server.
config.ts`, and `sentry.edge.config.ts` — before setting anything, rather
than assuming. Set it on Vercel (production + preview), forced a fresh
`vercel --prod` rebuild so it's actually baked in.

Verification needed a workaround: the codebase's own `/debug/sentry`
verification page (Slice 8a) is deliberately gated to 404 in production
(`NODE_ENV === "production"` → `notFound()`) — correct behavior, confirmed
it still 404s on the live site, but that also means it can't be used to
trigger a real production test error, and no browser was available in this
session to drive it in dev either. Sent a genuine test event directly to
Sentry's ingest API instead (the same mechanism Sentry's own "send test
event" UI uses under the hood) — `200 OK`, returned event ID matched what
was sent, confirming the DSN/project accepts events end-to-end. No Sentry
read access available to independently confirm the event rendered in the
Issues feed past ingestion acceptance — asked the user for one quick glance
to close that last gap, rather than claiming full confirmation without it.

### Slice 5 plan-first — parent-initiated Paystack payments (D4)

Given the "moves money, not just reads data" stakes, read the actual Phase 3
payment code before proposing anything: `PaymentsService.initPaystack`,
`handleWebhook`, `applyPaystackSuccess`/`Failed`, `verifyPaystack`, and
`PaystackController`. Key findings that shaped the plan:

- The webhook path is already 100% identity-agnostic — confirmed by reading
  it, not just trusting D4's own claim that no new webhook path is needed.
  It resolves everything from the `PSK-{schoolId}-{paymentId}` reference
  string; no `AuthContext` anywhere in that path.
- `recordedBy` on `Payment` is documented as "userId" specifically — not
  reusable for a guardian id. Resolved by writing `recordedBy: null` for
  guardian-initiated payments (no staff actor, same semantic the webhook
  path already uses) and recording *which* guardian in the audit log's
  `metadata` instead, mirroring `portal-auth.service.ts`'s own
  guardian-id-in-`audit_logs.user_id` precedent.
- The staff `initPaystack` call passes no `callbackUrl`, relying on
  Paystack's dashboard default (which points at `apps/web`) — wrong app
  entirely for a guardian. Guardian flow passes one explicitly, built from
  the existing `PORTAL_BASE_URL` env var.
- Checked `.env`'s `PAYSTACK_SECRET_KEY` — confirmed it's still the
  unfilled `.env.example` placeholder (`sk_test_replace_me`, byte-length
  match), not a real sandbox key. Non-issue: Phase 3's own
  `payments.service.spec.ts` already established the stub-at-consumer
  pattern for exactly this reason.
- Resolved "amount": server-computed as the exact outstanding balance,
  never accepted from the client — a deliberate divergence from the
  staff endpoint (which legitimately supports partial payments), making
  "you can only pay what's actually owed" a server-enforced invariant
  rather than a UI default.
- Found and flagged a real, pre-existing gap while reasoning through
  idempotency: neither `initPaystack` call site guards against a second
  in-flight PENDING payment on the same invoice, so two completed
  checkouts (double-click, two tabs) can both apply via the webhook and
  sum to a real overpayment. Proposed a narrow mitigation scoped to just
  the new guardian endpoint, explicitly not a fix to the shared webhook
  code — approved, with the instruction to also log it properly in
  `docs/deferred.md` once built (see below).

Plan approved as written.

  building it surfaced a real choice — `applyPaystackSuccess`/`Failed` are

  to *trigger* that, just observe it. Landed on a **pure read** — no
  Paystack API call, no apply logic — with the frontend polling every 2s

### Self-heal gap follow-up — pure read reversed, active-verify shipped

Before signing off, checked the two real questions the pure-read design's
own comment had assumed away rather than actually verified: what the
callback page shows once its poll window elapses, and whether staff have
any indirect way to unstick a guardian-initiated PENDING payment. Traced
both against the real code (not memory): the callback page's `PENDING`
branch silently stops polling with no actionable path forward, and
`FinanceService`/the debtor list have zero `PENDING`-status awareness — a
bursar can't see or manually resolve a stuck guardian payment through any
existing staff surface either. Neither held up as a real self-heal path,
so reversed the pure-read design per approved §3:

- `PaymentsService.verifyAndApply(schoolId, reference)` — new narrow
  public method, extracted from the old `verifyPaystack` (which is now a
  thin wrapper delegating to it). Both the staff endpoint and the new
  guardian endpoint call the same code, so there's exactly one "check
  Paystack, apply if not already applied" implementation, not two.
- `applyPaystackSuccess`/`applyPaystackFailed` hardened with an atomic
  `updateMany({ where: { id, status: "PENDING" } })` claim before doing
  any receipt/recompute/audit work — the loser of a concurrent race gets
  `count: 0` and bails before touching anything. This closes a real gap:
  `verifyAndApply` now has two independent callers (guardian poll,
  webhook) that can land concurrently, which the original single-caller
  design never had to defend against.
- `PortalPaymentsService.getByReference` (pure read) replaced with
  `verify()` — authorization (tenant + `withGuardian`) runs *before* any
  Paystack call, then delegates to `verifyAndApply`. A guardian polling a
  reference they have no business seeing never reaches Paystack.
- Frontend: `payments/callback/page.tsx` polls the same endpoint, but the
  semantics underneath changed materially — every poll is now a genuine
  Paystack check, so a response is always terminal (SUCCESS/FAILED) once
  it succeeds. There is no "still PENDING" response anymore, which made
  the old 20s/10-poll "still confirming" design dead code for its
  original purpose. Reworked: retries now back off *transient failures*
  (our fetch failing, or Paystack's API erroring) rather than an async
  webhook that might still land, 3s/5-attempt window (~15s), and a
  "Check again" manual retry button once that's exhausted — each click is
  a real independent verification attempt, not a hopeful re-read.
- New tests proving (not asserting) the concurrency claim: two
  simultaneous guardian polls of the same reference, and a guardian poll
  racing a direct webhook call for the same payment — both assert exactly
  one `payment.paystack-confirm` audit row and a non-doubled
  `invoice.totalPaid` afterward.

Full re-verification: `apps/api` typecheck/lint clean, full vitest suite

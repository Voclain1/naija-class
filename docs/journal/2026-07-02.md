### Plan-first — Phase 3 Slice 8 (Paystack integration)

Read `docs/modules/phase-3.md §8` and the existing payments module. Drafted and
locked 11 decisions into `docs/modules/phase-3.md §17`. Key choices:

| Decision | Locked |
|---|---|
| Initiation actor | Bursar-initiated (auth-gated init endpoint); frontend opens `authorizationUrl` in new tab |
| Customer email | Server-resolved (guardian lookup, then synthetic fallback) — not passed from frontend |
| Paystack reference format | `PSK-{schoolId}-{paymentId}` — embeds routing key to avoid cross-tenant lookup in webhook |
| Webhook public endpoint | Separate `PaystackController` (no class-level AuthGuard); `PaystackWebhookGuard` only |
| Raw body preservation | `NestFactory.create(AppModule, { rawBody: true })` + `req.rawBody` in guard |
| Webhook idempotency | Status check before processing + `@@unique([schoolId, paystackReference])` as DB guard |
| `charge.failed` | Update to FAILED, don't touch invoice totalPaid, always return 200 to Paystack |
| Verify self-heal | `GET /paystack/verify/:reference` applies same `applyPaystackSuccess()` helper as webhook |
| Shared helper | `applyPaystackSuccess()` private method used by both webhook and verify — no duplication |
| Permissions | No new strings; `init` reuses `payment.record`, `verify` reuses `payment.read` |
| Schema | No migration needed — all Paystack fields already exist on Payment model from slice 7 |

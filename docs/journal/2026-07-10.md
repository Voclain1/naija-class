### CP2 — Invite-flow role selector

User caught that `/staff/invite` still showed "Invite an administrator"
with no role dropdown — CP2 hadn't actually landed despite being scoped in
the plan-first. Built it:

- **`inviteAdminSchema`**: added `roleKey: z.enum(["admin", "bursar"]).default("admin")`.
  `InviteAdminInput` switched from `z.infer` to `z.input` for this schema —
  keeps `roleKey` optional in the TS type callers see (tests, the web form,
  direct service calls), so the ~14 existing call sites across
  `users.service.spec.ts` / `invitations.service.spec.ts` that predate slice
  15 didn't need touching. `UsersService.invite` defaults a missing
  `roleKey` to `"admin"` itself, so behaviour is identical whether or not a
  value passed through `inviteAdminSchema.parse()` first.
- **`UsersService.invite`**: re-validates `roleKey` against the `"admin" |
  "bursar"` allow-list server-side (throws `ValidationError
  INVALID_ROLE_KEY` otherwise) — defence-in-depth for direct callers that
  bypass `ZodValidationPipe` entirely, not just trusting the enum. Uses the
  resolved `roleKey` (not a hardcoded `"admin"` literal) when creating the
  invitation row and writing audit metadata.
- **`InvitationsService.accept`** needed zero changes — it already resolved
  the granted role generically by `row.role_key` (this is how `teacher`
  invitations worked too), so a bursar invitation flows through to a real
  bursar role grant automatically now that the role exists (from CP1).
- **`/staff/invite` UI**: added a native `<select>` Role dropdown

## Payroll CP3 — payroll run without money movement

Plan-first approved (payroll CP3/CP4, split at "money movement" — CP3 builds
the run with no Paystack transfer, CP4 adds the transfer). Built CP3 in full:

- **Schema**: `PayrollItem` model + migration (`payroll_items`, FORCE RLS,
  `PayrollStatus` enum DRAFT/APPROVED/PAID/FAILED — PAID/FAILED and
  `paystackTransferCode` reserved for CP4). `@@unique([schoolId, userId,
  period])` — one item per staff member per pay period.
- **RBAC**: `payroll.read`/`payroll.process` added to `PHASE_3_PERMISSIONS`
  and granted to bursar; `payroll.transfer` added but excluded from bursar
  (mirrors `payment.refund`/`staff-bvn.reveal` — the highest-trust
  money-movement mutation is admin+owner only). `KNOWN_BURSAR_EXCLUSIONS` in
  `permissions-coverage.spec.ts` updated so the close-out audit didn't just
  silently pass an unaccounted-for exclusion.
- **`PayrollService`**: create (netSalary computed server-side, `deductions`
  is a flat line-item list — NOT a Nigerian PAYE tax-bracket engine, per the
  plan-first's D3), update (DRAFT-only guard, mirrors GradingService's
  freeze-after-scores invariant and Invoice's snapshot-on-issue at the row
  level), approve (DRAFT→APPROVED, audited as `payroll.approve` — distinct
  from the `payroll.process` permission string gating it), generatePayslip
  (APPROVED-only guard — generating a payslip against numbers that can still
  change would produce a document that silently stops matching the row).
  Payslip ships as signed-URL HTML, exactly mirroring the payment-receipt
  precedent (plan-first D5) rather than a second Puppeteer/PDF pipeline.
- 15 new integration tests (`payroll.service.spec.ts`) + a new
  `PayrollController` coverage block in `permissions-coverage.spec.ts`.

**Real bug found via the manual gate, not a spec**: `DevStorageController`
(the dev-only signed-URL server for the filesystem storage driver) hardcoded
`Content-Type: application/pdf` and `Content-Disposition: attachment` for
*every* file it serves, because `FilesystemStorageDriver.put()` silently
discards the contentType/contentDisposition it's given (dev-only, no
metadata sidecar) and nobody had reason to look until now. This was already
silently wrong for payment-receipt and expense-receipt HTML — a signed
receipt URL forced a mislabeled-PDF download instead of rendering inline —
but every prior manual gate for those features checked byte-fidelity of the
stored file, never the Content-Type header the signed URL actually served.
Fixed by deriving both headers from the path extension (`.html` → `text/html`
+ inline; `.pdf` unchanged). `expense-receipt`'s extensionless path still
can't get a correct type this way (no sidecar to read from) — falls through
to `application/octet-stream` now instead of a wrong `pdf` label; tracked as
still-open in `docs/deferred.md`.

**Manual gate** (real HTTP, PowerShell against a compiled `dist/main.js` —
hit the same "bootstrap hangs after PartitionService, needs a kill+retry"
flakiness from Slice 15's gate again, resolved the same way): create a DRAFT
→ update deductions (netSalary recomputes 500k−40k=460k, then −50k=450k
across two edits) → approve (DRAFT→APPROVED) → generate payslip (200,
`Content-Type: text/html; charset=utf-8`, `Content-Disposition: inline` —
confirms the dev-storage fix) → attempt to edit the APPROVED item → 409.
All five steps green. Test schools cleaned up afterward.

Full verification: typecheck 12/12, lint 8/8, test 1147 passed (1 confirmed-
transient Puppeteer/Chromium launch timeout on re-run in isolation — 16/16
green; 2 pre-existing skips unrelated).

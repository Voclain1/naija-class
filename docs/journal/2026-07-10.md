### Plan-first: 5 open questions, 1 real discrepancy

Read `docs/modules/phase-3.md` §15 + `CLAUDE.md` and drafted the plan-first
before touching code, per the standing workflow. Two corrections surfaced
during research, both confirmed against the actual code rather than the
stated context:

- **`payment.refund` / `staff-bvn.reveal` are not owner-only.**
  `PHASE_3_OWNER_ONLY_PERMISSIONS` contains exactly one string
  (`auth.2fa.manage`). Admin already holds refund + BVN-reveal; bursar's
  exclusion is a separate, narrower allow-list problem, not an owner-only
  carve-out. Bursar's grants are built the same way `teacher`'s are — an
  explicit inclusion list (`PHASE_3_BURSAR_PERMISSIONS`), not "admin minus a
  few."
- **`billing.delete` doesn't exist anywhere in code.** `ARCHITECTURE.md` and
  early `phase-3.md` drafts named it as the owner-only "hard-delete of
  finance records" gate, but every actual finance hard-delete
  (fee-category/fee-item/expense-category/expense/payment-plan) already
  shipped as its own granular, **admin-accessible** permission across slices
  4/9/13 — each locked and explicitly tested as "no owner-only restriction."
  The placeholder was never wired to an endpoint.

Presented both resolutions (add it as an unused owner-only string vs. retire
the placeholder and document why); user picked **retire** — avoids dead code
and avoids reversing three already-shipped, already-tested decisions.

Also confirmed via `packages/db/src/seeds/system-roles.ts` that `teacher` was
introduced as pure data (no Prisma schema change) plus a companion idempotent
migration — bursar follows the identical shape. And confirmed
`docs/deferred.md` has nothing that blocks Phase 3 close-out (fresh re-read,
not from memory).

### CP2 — Invite-flow role selector

User caught that `/staff/invite` still showed "Invite an administrator"
with no role dropdown — CP2 hadn't actually landed despite being scoped in
the plan-first. Built it:

- **`inviteAdminSchema`**: added `roleKey: z.enum(["admin", "bursar"]).default("admin")`.
  `InviteAdminInput` switched from `z.infer` to `z.input` for this schema —
  keeps `roleKey` optional in the TS type callers see (tests, the web form,
  direct service calls), so the ~14 existing call sites across
  `users.service.spec.ts` / `invitations.service.spec.ts` that predate slice
  15 didn't need touching. `UsersService.invite` defaults a missing
  `roleKey` to `"admin"` itself, so behaviour is identical whether or not a
  value passed through `inviteAdminSchema.parse()` first.
- **`UsersService.invite`**: re-validates `roleKey` against the `"admin" |
  "bursar"` allow-list server-side (throws `ValidationError
  INVALID_ROLE_KEY` otherwise) — defence-in-depth for direct callers that
  bypass `ZodValidationPipe` entirely, not just trusting the enum. Uses the
  resolved `roleKey` (not a hardcoded `"admin"` literal) when creating the
  invitation row and writing audit metadata.
- **`InvitationsService.accept`** needed zero changes — it already resolved
  the granted role generically by `row.role_key` (this is how `teacher`
  invitations worked too), so a bursar invitation flows through to a real
  bursar role grant automatically now that the role exists (from CP1).
- **`/staff/invite` UI**: added a native `<select>` Role dropdown
  (Admin/Bursar, matching the existing shadcn-adjacent styling convention
  from the finance dashboard), wired to the form's `roleKey` field with a
  `admin` default. Title changed from "Invite an administrator" to "Invite a
  staff member"; the subtitle now reads "...as an admin"/"...as a bursar"
  depending on the live selection.
- Fixed stale comments in `staff-api.ts` and the page header that described
  the pre-slice-15 "hardcoded to admin, no dropdown" state.
- Updated the matching `docs/deferred.md` entry: **partially resolved** —
  admin/bursar now have a real dropdown; teacher stays deferred for its own
  reason (TeacherProfile fields aren't on the invite-accept path yet).

Added 6 new tests: bursar invite + default-to-admin + invalid-roleKey
rejection (service level), bursar invite + invalid-roleKey-via-HTTP
(controller level), and a bursar-accept test in `invitations.service.spec.ts`
that spot-checks the granted role's permission set (`invoice.read` present,
`staff-bvn.reveal`/`student.create` absent). Full re-verification: typecheck
12/12, lint 8/8, **1113 tests passed** (2 pre-existing skips).

**Manual gate (real HTTP, not vitest mocks)**: hit an environment snag first
— `pnpm dev`'s `nest start --watch` crashed on a stale `dist/main` race, and
a subsequent background launch hung with no `app.listen()` log at all
(traced to Windows Git Bash running out of fork resources — `cygheap`/fork
errors — from too many concurrent background processes in this session, not
a Docker or code problem; Docker itself was confirmed healthy throughout via
PowerShell). Killed the stuck process, relaunched `node dist/main.js`
directly via PowerShell `Start-Process`, and it came up clean. Then, against
that real running server:

- Signed up a fresh school, flipped it to `ACTIVE` directly in Postgres.
- `POST /users/invite` with `roleKey: "bursar"` as the owner → 201, correct
  invitation shape.
- `POST /invitations/:token/accept` → 200, real user created.
- As the new bursar's own session token: `GET /finance/debtors` → 200,
  `GET /expenses` → 200, `GET /invoices` → 200, `GET /finance/dashboard`
  (with a real term created via the owner token) → 200 with correct
  zeroed aggregates.
- `GET /users` (staff list) → 403, `GET /students` → 403 (non-finance,
  correctly denied).
- `POST /refunds` → 403 (`payment.refund` correctly excluded).
- `GET /users/:id/bvn/reveal` → 403 (`staff-bvn.reveal` correctly excluded).

Every check landed exactly where the RBAC design says it should. Cleaned up
the test school afterward (manual FK-ordered delete, since raw SQL doesn't
cascade the way Prisma's `.delete()` does in the test suite's `afterAll`).

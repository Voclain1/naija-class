  (Decision C), so email+password alone is ambiguous across schools; the

  `argon2.verify` loop, documented as interim (option ii).**

### Full implementation pass

With the migration approved, built the rest of the slice in one pass rather
than stopping again mid-way:

- `withGuardian` (`packages/db/src/with-guardian.ts`) — uses
  `@school-kit/types`'s `ForbiddenError`, not `@nestjs/common`'s
  `ForbiddenException` (packages/db has no NestJS dependency; phase-4.md's
  own sketch used a generic name that would have introduced one).
- `GuardianAuthGuard` + `createGuardianSession` + `GuardianAuthContext` +
  `CurrentGuardian` — mirror the staff `AuthGuard`/`sessions.ts` pair
  exactly. No endpoint uses the guard yet in this slice (all three portal
  endpoints are pre-auth by nature) — built now because phase-4.md assigned
  "resolve the guard mechanism" to slice 2, and a login endpoint with no way
  to ever verify the session it mints would be incomplete.
- `PortalAuthModule` (service + controller): `POST /portal/login` (the
  multi-candidate verify, timing-attack dummy-hash defense mirrored from
  `AuthService.login`, `ConflictError("AMBIGUOUS_GUARDIAN_ACCOUNT", ...)` on
  more than one match), `GET /portal/invitations/:token`,
  `POST /portal/invitations/:token/accept` (sets `passwordHash` +
  `emailVerified` on the *existing* Guardian row, unlike staff accept which
  creates a new User).
- `POST /guardians/:id/invite` added to the existing `GuardiansController`/
  `GuardiansService` (admin-facing, `@Permissions("guardian.invite")`) —
  console-logs the accept URL via `PORTAL_BASE_URL`, same Resend-deferred
  pattern `UsersService.invite` already uses for staff.
- `PHASE_4_PERMISSIONS` + `permissions-coverage.spec.ts` extension (pinned
  `invite` assertion, `PortalAuthController`'s three public handlers
  asserted `toBeUndefined()`, seeded-role-grant checks for owner/admin/
  bursar).
- Portal UI: `apps/portal/src/app/api/portal/[...portal]/route.ts` (proxy +
  `sk_portal_session` httpOnly cookie, `Domain=portal.schoolkit.ng` only
  when the request's own Host matches production — same conditional as
  slice 1's dev-cookie-check route), `login/page.tsx` wired with a real
  submit handler, new `invitations/[token]/page.tsx` accept flow, `page.tsx`
  (`/`) changed from an unconditional redirect to a session-cookie-presence
  check. No dashboard built (out of scope — slice 4's job); both success
  states are minimal inline placeholders.

### Two bugs caught by actually running the suite, not by typecheck

1. `admin`'s seeded permissions are a static array (`ADMIN_PERMISSIONS` in
   `system-roles.ts`) built from `PHASE_0..3_PERMISSIONS`, not derived from
   `ALL_PERMISSIONS` — adding `PHASE_4_PERMISSIONS` to the latter alone
   would never have reached the seed. Fixed by adding it to
   `ADMIN_PERMISSIONS` directly, plus an idempotent
   `UPDATE "roles" SET permissions = permissions || '["guardian.invite"]'...`
   appended to the migration for already-provisioned databases (same
   pattern as every prior RBAC-rollup migration).
2. ESLint's `basePrisma` allowlist (`packages/config/eslint/base.js`)
   correctly rejected both new pre-tenant call sites
   (`guardian-auth.guard.ts`, `portal-auth.service.ts`) until added, with
   the same justification-comment discipline as the existing staff entries.

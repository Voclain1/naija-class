### Phase 3 Slice 13 — Expense tracking

**Category decision (locked)**: `ExpenseCategory` table mirrors `FeeCategory`
(id, schoolId, name unique-per-school, active, timestamps — no description,
no createdBy). `Expense.categoryId` is a **plain FK, no DB constraint** —
same convention as `FeeItem`'s scope columns, the *opposite* of
`FeeItem.category` (which does get an enforced relation). Consequence: the
category delete-guard uses a manual `db.expense.count()`, not a Prisma
`_count` relation; category name resolution on the web UI is entirely
client-side (mirrors `teacher-assignments-section.tsx`'s `armName()`/
`subjectName()` pattern).

**CP1 — schema, services, storage plumbing**:
- Migration `20260709000000_phase_3_slice_13_expenses`: `expense_categories`
  + `expenses` tables, FORCE RLS, `expense-category.*`/`expense.*` permission
  grants to admin (no owner-only carve-out — unlike `payment.refund`/
  `staff-bvn.reveal`, this isn't a highest-trust surface).
- `StorageObjectKey` extended with `{ kind: "expense-receipt"; expenseId }`.
  Deliberately **extensionless** path (`schools/<id>/expenses/<id>/receipt`)
  — unlike `payment-receipt` (always `.html`), the upload can be
  JPEG/PNG/PDF; Content-Type is carried as S3/R2 object metadata from the
  upload's mimetype, not inferred from the URL. Avoids needing a new column
  just to reconstruct the same key on a later `get`/`signUrl`/`delete` call.
  One correction mid-review: `pathFor()`'s `schoolId` comes from the
  function's own parameter, not a `key.schoolId` field — no variant in the
  discriminated union carries `schoolId`.
- `ExpenseCategoryService` + `ExpenseService`, mirroring
  `FeeCategoryService`/`FeeItemService` CRUD shape exactly.
- 33 integration tests across both services.

**CP2 — controllers, receipt upload, web UI**:
- Extracted `UploadErrorFilter` out of `imports.controller.ts` into
  `common/upload-error.filter.ts` (parameterized by size-label string) —
  first controller-scoped Multer error filter needed a second consumer, so
  the duplication crossed the "extract it" threshold. Imports' own 3 upload
  endpoints switched to the shared filter; its test suite re-verified green
  (26+ tests untouched).
- `POST/GET /expenses/:id/receipt` — deliberately a **separate call** from
  create/update (D2 of the plan-first): mixing a multipart file with
  Zod-validated JSON in one request is the exact complexity NestJS's
  file-upload story handles badly. 8 MB cap (vs. CSV's 5 MB — phone camera
  photos routinely exceed that), MIME allow-list (jpeg/png/pdf).
  Re-upload replaces the previous file (delete-then-put); expense delete
  unconditionally calls `storage.delete()` (idempotent no-op if nothing was
  ever uploaded) so nothing orphans.
- `apiFetch` (web) extended to detect `FormData` bodies and skip both the
  JSON.stringify and the Content-Type override — it previously always
  stringified the body, which would have silently broken every multipart
  upload with no client-side error.
- Web UI: `/finance/expenses` as **one screen, not two** — expenses are a
  flat ledger (category is a tag, not a scope hierarchy the way `FeeItem`
  needs), so category management is a lightweight modal rather than
  earning its own `/settings/finance/...` route the way fee catalog did.
- `permissions-coverage.spec.ts` extended to 90 tests.

**Manual gate** (live dev server, real filesystem storage driver):
category → expense → upload receipt (`image/jpeg`) → signed URL round-trips
byte-exact (10/10 bytes) → edit amount → delete category blocked
(`EXPENSE_CATEGORY_HAS_EXPENSES`, 1 expense still attached) → upload
`application/zip` rejected (`INVALID_RECEIPT_TYPE`) → delete expense →
delete category succeeds → confirmed the receipt **file itself** (not just
the DB row) was gone from disk, not just orphaned.

**Full verification**: typecheck 12/12 packages, tests 1090 passed, lint
8/8 clean — twice, before and after the manual-gate session.

### Guardian invitation manual-test gap found

Arinzechukwu created guardians via the roster (manual form + 4-row CSV
import) and went looking for a "Send portal invitation" action in
`apps/web` — it doesn't exist. Confirmed by checking the Slice 2 commit
directly: `git show --stat 9e08a89` touches zero files under `apps/web`, and
no guardian-invitation reference exists anywhere in its source. This was
never in Slice 2's scope (backend + the *portal's* own login/accept pages
only) — not a bug, just an admin-side UI gap that's presumably a later
slice's job.

Gave him the manual trigger path instead: login → `GET /guardians?search=`
to find the target guardian's `id` (the roster UI doesn't surface raw IDs)
→ `POST /guardians/:id/invite` (no body) → the response's `acceptUrl` is the
real, live accept-invite link. Flagged one prerequisite: the endpoint 400s
`GUARDIAN_HAS_NO_EMAIL` if the guardian record has no email, which some
CSV-imported rows may lack depending on what the CSV had — worth checking
via the search response before assuming a null result is a bug. Also noted
`guardian.invite` is owner/admin only, not bursar.

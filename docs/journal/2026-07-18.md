## Work log

### Root cause: `NEXT_PUBLIC_API_URL` missing on the recreated Vercel project

Picked up where 2026-07-17 left off: Arinzechukwu's manual guardian-invite
walkthrough hit a generic "invitation not valid" message on the accept-invite
page. Traced it back to `school-kit-portal`'s Vercel project — it was deleted
and recreated during Slice 1 CP2 troubleshooting (to fix a stuck Root
Directory/Build Command setting) and silently came back with **zero**
environment variables. `NEXT_PUBLIC_API_URL` had been missing for 5+ days;
nothing caught it because the portal's `/api/health` check doesn't call the
API at all, so only a route that actually proxies to `apps/api` would ever
expose the gap. Confirmed via `vercel env ls` and set the var directly.

### PR #96 — portal proxy route stopped masking upstream failures

With `NEXT_PUBLIC_API_URL` unset, the proxy route's `fetch()` to the API
threw `ECONNREFUSED` uncaught. Next.js returned its own generic non-JSON 500,
and the frontend's error parsing fell back to a generic "invitation not
valid" — actively misleading, since the real problem was the route never
reaching the API, not a bad token. Fixed `apps/portal/src/app/api/portal/
[...portal]/route.ts`: wrapped the fetch + JSON.parse in try/catch, returning
the same `{ error: { code, message } }` envelope the rest of the app uses on
failure, so the frontend now surfaces `UPSTREAM_UNREACHABLE`/502 instead of
guessing. Also added `.vercel` to `.gitignore` (CLI hygiene, matching the
per-app entries already in `apps/web` and `apps/portal`), and backfilled the
2026-07-16 and 2026-07-17 journal entries with the full incident record
discovered while writing this fix.

### PR #97 — documented the failure category in CLAUDE.md

Wrote up the gotcha in CLAUDE.md's Environment variables section: recreating
a Vercel project does not carry over its env vars, confirmed against the
`school-kit-portal` incident above. Paired it with the `PORTAL_BASE_URL` Fly
secret miss from the Slice 2 migration pass (added to `.env.example`/CI,
never actually `flyctl secrets set` on the running `school-kit-api` app) as
the same failure category on the other platform: config landing in the repo
but never verified against the actual deployed environment. Added the
standing rule — always run `vercel env ls` right after recreating a project,
and don't trust "build succeeded" or "health check passed" as proof env vars
survived.

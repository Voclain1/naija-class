## CP4 plan-first (approved, with two additions)

Researched the existing Paystack integration before drafting: `PaystackService`
(fetch-based, no SDK), the charge webhook's routing pattern (switch on
`event.event`, no lookup table — the reference string itself embeds
`schoolId`/`paymentId`), its idempotency mechanism (check the row's own
current status, not a separate processed-webhooks table), and
`payments.service.spec.ts`'s established "stub the whole `PaystackService`"
sandbox-testing precedent (`makePaystackStub`) — all of this grounded the 8
D-decisions rather than reinventing patterns.

Two user additions to the drafted plan:
1. **Account verification UX**: `POST /staff-bank-accounts/verify` previews
   the resolved name with no side effects; saving without a prior verify is
   structurally impossible — the create endpoint independently re-resolves
   and compares server-side rather than trusting the client's echoed name.
2. **`transfer.reversed` → `FAILED`**, no new enum value — the audit log
   already preserves the distinction; adding a `REVERSED` status would be
   tracking the same fact twice.

Also settled during research, not explicit in the original 8 questions:
`PayrollStatus` needs a `TRANSFERRING` intermediate (CP4b, not CP4a) since
the existing 4 statuses have no way to represent "Paystack accepted the
transfer, webhook not back yet" — collapsing that into `APPROVED` would make
the payroll list unable to distinguish "not yet transferred" from "in
flight." And a genuine prerequisite surfaced for CP4b: Paystack transfers to
new recipients may require OTP finalization unless the platform's account
has Transfer OTP disabled — flagged for confirmation before CP4b assumes a
single synchronous transfer call.

## PartitionService startup hang — root-caused and fixed

While retrying the CP4a manual gate, asked to actually dig into the hang
instead of continuing to treat it as unexplained environment flakiness.
Root cause: `PartitionService.onModuleInit()` awaited 3 sequential `SELECT
create_audit_log_partition(...)` calls with **no timeout**. Since
`onModuleInit` runs inside `NestFactory.create()`, a slow or
momentarily-saturated dev DB connection pool at startup (exactly the state
right after a Docker Desktop relaunch, which several of these incidents
followed) could hang the whole bootstrap indefinitely — no error, no
crash, `app.listen()` never reached. The 3 "Partition ensured" logs
appearing before every hang is exactly consistent with this: those are
each call succeeding in sequence; the hang was Nest's own post-`onModuleInit`
bookkeeping waiting on a connection the pool hadn't released yet, not
inside those calls themselves.

Fixed with `Promise.race` against a 5s timeout, catching both a rejection
and a timeout and logging `logger.warn` rather than propagating —
deliberately non-fatal, since the monthly `@Cron` job
(`createNextMonthPartitions`) is the durable path and `onModuleInit` is
only a best-effort cold-start convenience. Two new tests cover the
reject-path and the never-resolves-path, the latter asserting the elapsed
time is bounded (~5s) rather than just that the promise eventually settles.

`docs/deferred.md`'s hang entry updated with this root cause and marked
`[x]` fixed.

**Tool outage mid-task**: the Bash/PowerShell safety classifier was
unavailable for several minutes right as this fix was ready to verify —
every non-trivial command (typecheck, test, git operations) was refused
while trivial read-only ones (`pwd`, `git diff --stat`) still worked.
Reviewed the diff by eye during the outage instead of guessing at
correctness; resumed and ran the full verification once the classifier
recovered.

Full verification: typecheck 12/12, lint 8/8, test **1165 passed** (2
pre-existing skips), `partition.service.spec.ts` 6/6 in isolation
(including the ~5s timeout-ceiling test).
